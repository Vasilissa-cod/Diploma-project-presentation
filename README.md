### 1.1. Цель работы

Целью данной дипломной работы является проведение тестирования приложения FatSecret для обеспечения качества программного продукта и выявления потенциальных дефектов [1, с. 15].

### 1.2. Задачи работы

Для достижения поставленной цели необходимо решить следующие задачи:

- изучить тестируемое приложение и его предметную область;
- провести анализ функциональных и технических свойств приложения FatSecret;
- определить методологии тестирования программного обеспечения [1, с. 45];
- определить технические средства тестирования [2];
- составить план тестирования, определить тестовые случаи;
- выполнить тестирование, исходя из составленного тест-плана [1];
- составить отчёт по произведённым тестам.

### 1.3. Актуальность темы

В современном мире много мобильных и веб-приложений, в том числе, для здоровья и фитнеса [3, с. 120]. Эти приложения являются многопользовательскими и доступны в сети Интернет, что повышает риски, связанные с их безопасностью и надёжностью. Для своевременного предупреждения и обнаружения проблем, такие приложения подвергаются различным видам тестирования: функциональное, нефункциональное, регрессионное и т.п.

Своевременное обнаружение дефектов способствует уменьшению временных затрат и экономии средств.

Тестируемое приложение FatSecret имеет огромное количество пользователей, в т.ч. среди категории фудтрекеров. Оно предназначено для отслеживания питания и подсчёта калорий.

## 2. АНАЛИТИЧЕСКАЯ ЧАСТЬ

### 2.1. Описание приложения FatSecret

FatSecret - это бесплатное приложение для отслеживания питания и подсчета калорий, доступное как в веб-версии, так и в виде мобильных приложений [7]. Приложение помогает пользователям контролировать питание, отслеживать физическую активность и достигать целей по весу. Компания Fatsecret была основана в 2006 году в Мельбурне, Австралия, Ленни Мозесом и Родни Мозесом. FatSecret работает на множестве языков, включая русский, английский, испанский, французский, немецкий, китайский и японский, поддерживая более 20 языков, что делает его доступным для пользователей по всему миру. Он доступен в виде приложения для iOS и Android, а также имеет веб-версию, позволяющую отслеживать питание, физические нагрузки и вес.

**Основные платформы веб- приложения:**
- Веб-приложение: https://foods.fatsecret.com/

### 2.2. Функциональные возможности

Основной функционал веб- приложения FatSecret включает  следующие возможности:

#### 2.2.1. Дневник питания
- Добавление продуктов и блюд
- Поиск по базе данных продуктов [7]
- Создание собственных рецептов
- Отслеживание потреблённых калорий

#### 2.2.2. Отслеживание веса
- Ведение журнала веса
- Графики изменения веса
- Установка целей по весу

#### 2.2.3. Дневник упражнений
- Добавление физических упражнений
- Подсчёт сожжённых калорий
- База данных упражнений

#### 2.2.4. Социальные функции
- Создание профиля
- Общение с другими пользователями
- Группы поддержки

### 2.3. Технические характеристики

- Поддерживаемые браузеры: Chrome (я использовала в работе), Firefox, Safari, Edge [1]
- Адаптивный дизайн для различных устройств [9, с. 89]

### 2.4. Требования к приложению

Определены следующие функциональные требования:

**1: Регистрация и авторизация пользователей**
- Регистрация через email

**2: Управление дневником питания**

**3: Поиск продуктов**
- Поиск по названию

## 3. ТЕСТ-ПЛАН

### 3.1. Стратегия тестирования

Для тестирования приложения FatSecret мной использовалась стратегия, основанная на современных подходах к обеспечению качества программного обеспечения и рекомендациях по веб-производительности:

**Что и как тестировать:**

В первую очередь, нужно было проверить основные функции — вход в аккаунт, работу дневника питания, поиск продуктов. Эти сценарии пользователи выполняют каждый день, поэтому здесь я использовала автоматизированные тесты на Java с Selenium WebDriver и JUnit 5. Автоматизация позволяет быстро прогонять одни и те же проверки после любых изменений.

Проверку на разных браузерах и устройствах, а также безопасность передачи данных проводила вручную и это не отражено в работе.

**Инструменты:**

Для организации процесса использовала Miro — удобно держать всё в одном месте и видеть общую картину. Для анализа производительности страниц использовался Lighthouse (Встроен в Chrome DevTools) также делалось мной вручную и не отражено в работе.

### 3.2. Уровни тестирования

Уровни тестирования ПО (классическая пирамида) включают Модульное (Unit), Интеграционное (Integration), Системное (System) и Приемочное (Acceptance) тестирование, которые проверяют ПО от отдельных частей до полностью собранной системы.

#### 3.2.1. Модульное (Unit) тестирование
- **Что:** Тестирование самых маленьких изолированных частей кода (функции, методы, классы)
- **Цель:** Проверить, что каждый компонент работает корректно сам по себе
- **В данной работе:** ⚠️ Не применялось (я тестировала готовое веб-приложение, а не исходный код)

#### 3.2.2. Интеграционное (Integration) тестирование
- **Что:** Тестирование взаимодействия между компонентами системы
- **Цель:** Проверить корректность обмена данными между модулями
- **В данной работе:** ⚠️ Не применялось (я не тестировала API)

#### 3.2.3. Системное (System) тестирование
- **Что:** Тестирование полностью собранной системы как единого целого
- **Цель:** Проверить соответствие системы функциональным и нефункциональным требованиям
- **В данной работе:** ✅ Применялось

**Автоматизированные системные тесты веб-интерфейса (5 тестов):**
1. TC001: Авторизация пользователя негативный тест
2. TC001: Авторизация пользователя позитивный тест
3. TC002: Поиск продуктов
4. TC003: Добавление продукта в дневник и проверка подсчёта калорий
5. TC004: Добавление записи в журнал

Технологии: Java 11, Selenium WebDriver,Selenide, JUnit 5, Allure Report.

#### 3.2.4. Приемочное (Acceptance) тестирование
- **Что:** Тестирование системы с точки зрения конечного пользователя
- **Цель:** Проверить соответствие требованиям заказчика и готовность к релизу
- **В данной работе:** ✅ Применялось — мануальное тестирование пользовательских сценариев

**Мануальные тест-кейсы (10 тестов, 90% успешно):**
- Функциональное тестирование: 9 тест-кейсов
- Тестирование производительности: 2 тест-кейса
- Тестирование безопасности: 1 тест-кейс

### 3.3. Типы тестирования

В работе будут применены следующие типы тестирования:

#### 3.3.1. Функциональное тестирование
- Smoke testing - проверка базовой функциональности
- Sanity testing - проверка основных функций
- User Acceptance Testing - проверка соответствия требованиям

#### 3.3.2. Нефункциональное тестирование
- Usability testing - тестирование удобства использования интерфейса
- Performance testing - проверка производительности приложения
- Security testing - проверка безопасности передачи данных
